# GTNH OC Ore Processing Automation System

本项目是一个专门为 **GTNH** 整合包设计的 **OpenComputers** 智能化矿物自动化处理方案。它采用 **控制中心（Server） - 终端（Client/Worker）分布式架构**，通过无线网络（Modem）动态控制多个 ME 接口（ME Interface）与数据库组件，实现全自动、按需分配、动态流转的复杂矿物处理链（如：粉碎 -> 水洗 -> 热离 -> 筛选 -> 离心等）。

相比于传统的固定 ME 标记（Pattern）或大量硬管道，本系统具有免去频繁铺设样板、动态负载均衡、避免机器死锁、支持高复杂逻辑过滤等绝对优势。

本方案配套 [矿处流程编辑器][OreEditor]([仓库地址][OreEditorRepo]) 允许在网页中离线编辑你自己的矿处流程, 然后倒入到你自己的服务器中。

## 🛠️ 系统核心架构

系统主要分为三个部分：**控制中心（Server）**、**工作终端（Client/Worker）** 以及 **公共逻辑库（Common）**。

```txt
          +-----------------------------+
          |        Server 控制中心      |
          |  (中央配置、工艺生成、菜单) |
          +--------------+--------------+
                         |
                         | 无线广播 (Modem)
                         v
            +------------+------------------+
            |                               |
            v                               v
+------------------------+      +------------------------+
|    Client 工作终端 01  |      |    Client 工作终端 02  |
| (负责：水洗/粉碎 机器) |      | (负责：热离/离心 机器) |
+-----------+------------+      +-----------+------------+
            |                               |
动态配置    |                               | 动态配置
            v                               v
+-----------+------------+      +-----------+------------+
| ME Interface + Database|      | ME Interface + Database|
+------------------------+      +------------------------+

```

### 1. Server 控制中心

* **中央调度**：运行在主控电脑上，通过交互式菜单（基于 `list_reader`）引导玩家设定矿物工艺路线、黑白名单等。
* **动态规则生成器 (`rule_spawner.lua`)**：根据玩家为某种矿石（如 Copper、Platinum）指定的处理流程，结合 `interface_type.lua` 中的命名模板，自动推导并生成每一个处理步骤终端所需的物品标签白名单，并通过网络下发。

### 2. Client 工作终端

* **分布式执行**：每个处理机器集群（或单个机器）连接一台终端电脑。终端包含 `数据库升级` （T1 即可）和 `适配器`(适配器需要与 `ME 接口` 相连接)。
* **智能对齐与标记**：终端接收到服务器的规则后，会持续扫描 ME 网络中的物品。一旦发现符合自身“职责”的矿物，会将其按数量从大到小排序，并将前 9 种物品的信息动态写入 ME接口, 自动供应这些矿物进入机器。
* **非阻塞型睡眠**：系统通过动态自适应调整 `sleepTick`，在没有矿物处理时自动延长休眠时间，有工作时迅速响应，极大降低服务器 Ticking 负担。

### 3. 高级逻辑引擎 (`logical_rule` & `logical_exectors`)

* 系统内置了一个基于 **AST（抽象语法树）** 的词法/语法解析器，支持复杂的布尔逻辑规则表达式（`&&`, `||`, `!`）以及括号优先级。
* 可以通过逻辑规则扩展条件，例如结合 ME 网络中的流体存储量、特定物品数量，甚至支持动态执行内联 Lua 脚本来进行超高级的流向阻断与条件触发。

```📂
├── run-server.lua          # Server 启动入口
├── run-client.lua          # Client 启动入口
├── config.lua              # 全局核心配置文件（端口、休眠时间、日志等）
├── cpy.lua                 # 运维刷机工具，用于快速将代码同步到其他挂载盘
├── ore.config              # 服务器保存的核心配置文件（持久化存储工艺路线与规则）
│
├── 📂 server/              # 服务器模块
│   ├── server.lua          # 服务器主程序及 CLI 菜单循环
│   ├── setting.lua         # 配置文件的读取、转换与持久化管理
│   ├── configurate.lua     # 职责类型（Role）与接口地址绑定交互
│   ├── role_chooser.lua    # 工艺流程选择器（合法性校验）
│   ├── rule_spawner.lua    # 核心算法：将工艺链膨胀为全套终端过滤规则
│   └── interface_type.lua  # 格雷矿物命名的基础模板定义（矿石、粉碎、洗净、粉等）
│
├── 📂 client/              # 客户端终端模块
│   ├── client.lua          # 客户端主网络事件监听（接收更新、启停指令）
│   └── worker.lua          # 终端工作核心：扫描ME、排序物品、映射数据库
│
└── 📂 common/              # 跨端公共库
    ├── list_reader.lua      # 基于 GPU/键盘事件的通用交互式列表渲染器
    ├── item_util.lua        # 物品 ID 字符串与 Table 的解析/序列化工具
    ├── logical_rule.lua     # 高级规则的词法解析与 AST 状态求值器
    ├── logical_exectors.lua # 逻辑执行器（提供物品标记、流体读取、内联Lua等元操作）
    └── debug.lua            # 分级日志打印与持久化工具

```

## ⚙️ 硬件需求

### Server 控制中心

* **电脑级别**：最低可以使用 Tier 1 电脑机箱, 推荐 Tier 2 电脑机箱(两个硬盘位置方便刷机)。
* **必备组件**：Tier 1 CPU、内存（规则少的话可以使用 2x Tier 1 内存）、显卡、屏幕、键盘、硬盘（带 OpenOS）。
* **网络组件**：T1/T2无线网卡或普通网卡（视距离而定）。

### Client 工作终端 (每组机器一个)

* **电脑级别**：最低可以使用 Tier 1 电脑机箱。
* **必备组件**：CPU、内存（规则少的话可以使用 1x Tier 1 内存）、T1/T2无线网卡或普通网卡、硬盘（带 OpenOS）。
* **核心组件**：
  * **T1数据库升级**：用于临时存放筛选出的物品快照。
  * **适配器**：紧贴 **ME接口**(可以贴住若干ME接口)，用于通过代码控制ME接口的供应槽。

## 🚀 安装与部署

1. **分发代码**：
将本项目的所有代码上传到主控电脑中。你可以使用附带的 `cpy.lua` 工具将代码批量复制到其他挂载的节点盘中。

```bash
# 示例：将 /mnt/abc 的代码作为 client 同步到 /mnt/def 盘，并写入自动启动项
cpy client abc def
```

1. **配置全局参数**：
修改根目录下的 `config.lua`，确保全网的 `network.port` 保持一致，并根据服务器配置决定是否开启 `debug` 日志。
2. **启动中央服务器**：
在主控电脑上运行：

```bash
run-server.lua
```

1. **启动工作终端**：
在每个终端电脑上运行：

```bash
run-client.lua
```

## 📖 使用指南与控制台菜单

启动服务器 `run-server.lua` 后，你将看到如下交互式命令行菜单：

```text
1. 设定矿处工艺    2. 开始筛选
3. 停止筛选        4. 配置接口
5. 查看矿处工艺    6. 更新终端Woker
7. 更新终端客户端  8.上传文件
9. 设置物品黑名单  10. 设置物品白名单
11. 重新搜索节点   12. 查阅现有节点
> 
```

此时输入某一项编号并回车即可进入相应的流程中.

### 1. 配置机器与接口 (首次运行必做)

**该流程会在识别到未知角色的ME接口时自动进入.**

* 选择 **`11. 重新搜索节点`**，服务器会广播寻找全网在线的 Client 终端。
* 选择 **`4. 配置接口`**，为搜索到的每一个 Client 端（以 ME Interface 的地址标识）分配其**职责（Role）**。
* 例如：将某个地址配置为 `水洗` 职责，然后再选择该职责对应的格雷机器类型为 `洗矿机`。系统会自动将其与 `interface_type.lua` 的模板关联。

### 2. 设定矿物工艺路线

* 选择 **`1. 设定矿处工艺`**。
* 输入或选择你想要自动化的矿物流（例如：`Copper`）。
* 接下来通过列表选择该矿物要流经的职责链。例如，依次选择：`粉碎` -> `水洗` -> `热离` -> `筛选` -> `离心`。
* 系统在保存时会调用 **工艺链校验机制**，确保当前机器能够承接上一个机器的产物输出（防止路线逻辑错误）。

### 3. 一键启停

* 选择 **`2. 开始筛选`**：服务器会根据所有工艺自动膨胀生成所有终端的 Label 白名单，并通过无线网络下发给各节点，终端立即开始在 ME 网络中抓取对应矿物并推入机器。
* 选择 **`3. 停止筛选`**：全网终端暂停扫描，安全挂起。

### 4. 远程运维

* 如果你修改了 `client/worker.lua` 或 `client/client.lua` 的本地代码，无需跑到每个机器前拷贝。
* 直接在服务器选择 **`6. 更新终端Woker`** 或 **`7. 更新终端客户端`**，服务器将无线广播最新的代码文件，所有终端接收后会自动覆写本地文件并**热重启**。

## 💡 高级功能：高级逻辑表达式 (Logical Rules)

除了基础的矿石名称膨胀规则外，本系统还支持极其强大的条件流控引擎。可在服务器通过配置加入特定角色的 `logicalRules` 中。

### 语法通式

支持布尔逻辑 `&&` (与), `||` (或), `!` (非) 以及小括号 `()`。

### 内置核心命令

#### **标记物品标签 (`mark-item-label` / 简写 `MIL`)**

* `{MIL: Purified Copper Ore}`
* *作用*：主动强制向此终端的筛选白名单中注入特定的物品标签。
* *返回值*: 一定返回 `true`

#### **标记物品ID (`mark-item` / 简写 `MI`)**

* `{MI: gregtech:gt.metaitem.01:2528}`
* *作用*：主动强制向此终端的筛选白名单中注入`黄石榴石粉`。
* *返回值*: 一定返回 `true`

#### **检查物品ID数量 (`check-item` / 简写 `CI`)**

* `{CI: Thaumcraft:ItemResource:14 < 1024} && {MI: gregtech:gt.comb:25}`
* *作用*：当ME库存中的 `世界盐` 数量少于 `1024` 个, 则主动强制向此终端的筛选白名单中注入`世界盐蜂窝`。
* *返回值*: 如果物品数量满足表达式则返回 `true`, 反之返回 `false`

![OreEditorLogicalCI]

#### **检查流体ID数量 (`check-fluid` / 简写 `CF`)**

* `{CF: chlorine <= 100000000} && {MI: gregtech:gt.metaitem.01:2944} && {mark-item-label: Salt}`
* *作用*：当ME库存中的 `氯` 数量少于 `100M` ml, 则主动强制向此终端的筛选白名单中注入 `岩盐` 及 `盐`。
* *返回值*: 如果流体数量满足表达式则返回 `true`, 反之返回 `false`

![OreEditorLogicalCF]

#### **打印信息 (`print` / 简写 `P`)**

* `{P: abc}`
* *作用*：在终端中打印 `abc`.
* *返回值*: 一定返回 `true`

#### **执行自定义 Lua 脚本 (`eval-lua` / 简写 `L`)**

* `{L: print('abc')}`
* *作用*：在终端中打印 `abc`.
* *返回值*: 当 Lua 脚本附带返回值且返回值为 `boolean` 类型时, 这个返回值将作为这个单元的返回值; 否则若 Lua 脚本执行成功才返回 `true`, 执行失败返回 `false`.
